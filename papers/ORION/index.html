四个核心 insight

语义空间连接动作空间

不让 LLM 打印浮点坐标，而是把特殊 token hidden state 当作可微接口，既保留推理能力，又避免文本数值解析不稳定。

无显式 BEV 的 3D 建模

像素沿 64 个 LID 深度 bin 反投影，生成 3D positional embedding；PETR query 再对多相机特征做 cross-attention。

时序不是堆叠多帧

queue_length=1。跨帧信息存在 query memory 中，并通过 ego motion 变换到当前坐标系。

端到端不等于直接控制

神经网络预测 waypoint，闭环执行仍由 PID 完成。这是学习规划与经典控制的分层系统。

重要纠偏：源码中没有独立的 BLIP-2 式 Q-Former。pretrain_qformer 是权重目录名；视觉桥接由 PETR extra query、scene memory 和 256→4096 投影承担。

多任务损失

感知与地图

loss_cls、loss_bbox、DN query、交通灯状态、地图类别与 11 个控制点回归。

运动与规划

agent 多模态运动、ego L1、地图边界、碰撞约束和 VAE prior/posterior 分布对齐。

为什么 VAE 合理

训练：
q(z | ego_feature, GT_future)  ← posterior
p(z | ego_feature)             ← prior
ProbabilisticLoss 对齐两者
z + ego_feature → GRU → 6 modes × 6 delta waypoints

推理：
只能访问 p(z | ego_feature)
按 ego_fut_cmd 选择一个 mode
对 delta waypoint 做 cumsum → 绝对轨迹

同一个场景可能存在多条合理轨迹。显式分布与命令模态能够避免直接回归平均出一条“不左不右”的轨迹。

真实的时序机制

连续 sampler

同一 batch slot 连续读取同一路线帧，使 memory 能够继承。

坐标补偿

ego_pose_inv 把历史 reference point 与 pose 变换到当前帧。

Memory 更新

每帧选择 top-300 proposal，前插到 600-slot memory。

queue_length=1 表示 backbone 只处理当前帧；跨帧信息存在 query memory，而不是图像维度。它节省显存，但依赖序列顺序和正确 reset。

闭环执行链

CARLA: 6 RGB + GPS + IMU + speed
  → RoutePlanner: command + near waypoint
  → ORION: 6 个 ego-frame waypoint
  → PIDController
  → carla.VehicleControl(steer, throttle, brake)
控制计算工程限制
Steering在预测折线上选择约 3.5m aim pointroute target 默认不直接参与 steering
Desired speed由前两个 waypoint 距离估计目标速度过低或超速比 >1.1 时刹车
Throttle / brake互斥处理throttle 上限 0.75
Speed cap固定规则speed > 5m/s → throttle=0
ORION 是端到端轨迹规划系统，但不是严格 sensor-to-control：最后一段仍是 PID 和规则。

开环好不等于闭环好

开环预测错误不会改变下一帧；闭环中一次偏差会改变后续相机画面，形成 covariate shift，并经 PID、车辆动力学和交通交互持续放大。因此 L2/collision 与 Driving Score/Success Rate 衡量不同能力。

高频盘问与回答骨架

ORION 相比传统端到端规划的关键创新是什么？

把检测/地图的结构化 token 注入 VLM，并用 <waypoint_ego> hidden state 驱动连续 planner，实现语义推理与动作空间的可微对齐。

QT-Former 在代码哪里？

没有独立 BLIP-2 式 Q-Former；实际由双 PETR head 的 extra query、时序 scene memory 和 256→4096 投影承担视觉聚合与桥接。

没有 BEV，如何获得 3D 几何？

利用相机内外参把像素沿 LID 深度 bins 反投影为 3D positional embedding，query 再对多视角 token 做 PETR cross-attention。

为什么检测和地图分两个 head？

对象与道路拓扑具有不同的 query、匹配和 loss；分头保留任务归纳偏置，最终在 LLM 前融合为统一视觉上下文。

为什么是六个规划 mode？

与 CARLA 六类 route command 对齐。网络同时预测六条，再按 command 选择，缓解多模态未来的均值化。

Stage 2 与 Stage 3 有什么区别？

架构相同；Stage 2 是 planning-only，Stage 3 以 80/20 混合 planning 与通用 QA，避免语言推理能力退化。

ORION 是真正端到端吗？

视觉到轨迹是联合优化的，但轨迹到控制由 PID 完成。因此它是 end-to-end planning，而不是纯 sensor-to-control。

你认为最大的工程风险是什么？

坐标系一致性、时序 memory reset、Stage checkpoint 手工串联、闭环 JPEG 域偏移、command mode 选择错误，以及 PID 启发式不匹配。

回答任何架构题的四段式

1. Problem

原方法具体卡在哪里？不要先背模块名。

2. Design

新方法加入了什么接口或归纳偏置？

3. Evidence

给出 tensor shape、loss 或源码调用点。

4. Trade-off

讨论显存、延迟、域偏移、可解释性与失败模式。
