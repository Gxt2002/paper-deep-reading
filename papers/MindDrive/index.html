三个最重要的 insight

RL 只学“做什么”

Decision Expert 只优化速度 meta-action；连续轨迹生成能力主要由 IL 固定下来。

双 LoRA 隔离能力

稀疏 reward 更新 Decision Expert，不破坏 Action Expert 已学会的轨迹执行能力。

Path 不参与 PPO

横向 path mode 来自 RoutePlanner。RL 主要改善纵向决策，而不是完整规划。

双 LoRA 专家

Decision Expert

读取 meta-action QA，在 7 个速度 token 上形成策略分布。Rollout 采样，评测 argmax，PPO 更新该 LoRA。

Action Expert

读取规划 QA，在 <waypoint_ego> 与 <path_waypoint_ego> 位置提取 hidden state，再由 VAE 生成轨迹。

Minddrive.forward_train()
└─ forward_pts_train()
   ├─ map_head → map query / lane prediction
   ├─ pts_bbox_head(outs_lane) → det query / agent motion
   ├─ concat vision tokens → Qwen2
   ├─ action_expert → two waypoint hidden states
   ├─ speed VAE/GRU → (B,7,6,2)
   ├─ path VAE/GRU  → (B,6,20,2)
   └─ perception + VLM + planning losses
架构细节：训练中 map head 先于 det head，det head 会接收 outs_lane；这比 ORION 中简单并列调用体现了更强的地图→运动耦合。

IL 损失如何耦合

感知与地图

检测 focal/L1、DN、交通灯、地图分类与 11 控制点回归、agent 多模态运动。

Speed / Path planning

选中 mode 的 L1、地图边界、agent 碰撞和 VAE prior/posterior 对齐；path 分支有对应的 pw_* loss。

IL 到 RL 的权重桥接

Stage 3 checkpoint
  ├─ action_expert LoRA → decision_expert LoRA
  └─ lm_head            → value_net

目的：让 policy 和 critic 从已学会的视觉语言表示开始，
而不是在昂贵的 CARLA 环境中随机探索。

README 明确说明 IL 阶段主要训练 Action Expert LoRA。复制 checkpoint 是在线 RL 能够稳定启动的关键，而不是普通格式转换。

Rollout → Buffer → PPO

CARLA observation
  → Decision Expert: π(a|s), a ∈ {0,...,6}
  → sample speed action + value V(s)
  → Action Expert: speed/path trajectories
  → PID → CARLA
  → sparse reward
  → RolloutBuffer
  → GAE
  → cached inputs_embeds PPO replay

PPO 目标

Policy ratio

r = exp(logπ_new - logπ_old)，clip 范围为 1±0.2。

Value loss

0.5 × MSE(V_pred, return)，由 value projection 预测。

GAE

γ=0.99, λ=1.0。稀疏终止奖励下接近 Monte Carlo，方差较高。

奖励

事件RewardEpisode
路线完成 100%+1结束
碰撞、闯灯、偏航、驶出车道等-1结束
普通中间 step0继续
关键风险：稀疏 ±1 reward、gae_lambda=1 和固定 rollout 多 epoch 训练会产生高方差与 policy staleness。PPO clipping/KL 只能缓解，不能消除该问题。

真正被 RL 更新的参数

主要只有 decision_expert LoRA 与 value_net_pro。视觉 backbone、PETR、Action Expert 和 trajectory decoder 冻结。因此这是“语言决策层 RL”，不是全链路端到端 RL。

字段形状语义

离线 annotation 是 10 Hz，CARLA rollout/eval 是 20 Hz；轨迹监督按 5 帧采样得到 0.5 秒间隔。时间戳和 memory 对齐错误会产生隐蔽的动态偏差。

轨迹到控制

Path waypoint：20×2

PID 在路径折线上选择约 3.5 米前方 aim point，用于 steering。

Speed waypoint：6×2

PID 根据前几个 waypoint 距离估计 desired speed，用于 throttle/brake。

系统还有 throttle 上限、油门刹车互斥和 speed > 5m/s → throttle=0 等工程启发式。因此 MindDrive 仍是 VLA planning + classical control。

工程 failure modes

Adapter 错误

Decision/Action adapter 激活错误会导致动作概率或轨迹 hidden state 失效。

Path command 错位

速度决策正确，但 RoutePlanner command 对错 mode 时仍会走错路径。

坐标与固定外参

CARLA 左右手系、Y 轴符号、compass 和硬编码投影矩阵容易产生静默偏移。

缓存 rollout 过旧

固定数据训练多个 epoch 后不再严格 on-policy，需要重新采样。

高频盘问

为什么不直接对连续轨迹做 RL？

动作空间巨大且 CARLA rollout 昂贵。7 类语言动作显著提升探索效率，连续轨迹交给 IL 训练好的 Action Expert。

双 LoRA 为什么优于两个完整 LLM？

共享基础视觉语言表示、参数增量小，同时隔离决策 RL 与轨迹执行，减少 catastrophic forgetting。

奖励如何回传到语言空间？

Rollout 保存速度动作、旧 log-prob、advantage 和缓存 embedding；PPO 对 7 个 meta-action token 的概率更新 Decision Expert LoRA。

为什么 path 不参与 PPO？

利用 RoutePlanner 强先验并减少组合动作空间；代价是 RL 不能优化横向选择和变道时机。

为什么要 convert checkpoint？

IL 主要训练 Action Expert。复制给 Decision Expert 与 value net，避免 policy/critic 从随机参数开始在线探索。

缓存 inputs_embeds 的 trade-off？

避免 PPO 重跑 EVA-ViT/PETR，速度快；但视觉不接收 RL 梯度，且缓存占磁盘，数据会随 policy 更新变旧。

GAE λ=1 在稀疏 reward 下意味着什么？

接近 Monte Carlo advantage，偏差低但方差高。可考虑更小 λ、progress shaping 或更密集的安全奖励。

MindDrive 真的是 on-policy PPO 吗？

目标是 PPO，但实现先保存 rollout，再用固定数据训练多个 epoch。数据逐渐 stale，必须依赖 clipping/KL 并定期重新 rollout。

0.5B 为什么能接近甚至超过 ORION-7B？

闭环性能不只由参数量决定。在线 reward 可以修正 IL 的决策偏差，有限动作空间也让小模型高效探索。

最大的工程风险是什么？

adapter 切换、route/path mode 错位、稀疏 reward、10/20 Hz 时间差、坐标外参、checkpoint 命名和 PID 启发式。
